# 屏幕亮度实时调节设计

**日期：** 2026-04-07

## 目标

为网页端增加一个可调亮度的设置项，用户拖动滑杆时立即同步到当前设备；设备端在接收亮度变更后立即生效，并将亮度持久化保存，重启后保持上次设置。

## 范围

本次改造包含以下内容：

- 网页设置弹窗新增「屏幕亮度」滑杆与百分比显示
- 亮度滑杆在拖动过程中即时同步到已连接设备
- 前端新增亮度状态缓存、读取与同步逻辑
- BLE 协议新增亮度设置与亮度查询命令
- WiFi 备选链路新增亮度设置与亮度查询接口
- 固件新增亮度读写、运行时应用和 NVS 持久化
- 固件启动时恢复上次保存的亮度值

## 非目标

本次不做以下内容：

- 不改造现有页面整体视觉风格
- 不新增自动亮度、环境光感应或亮度曲线功能
- 不重构现有图像发送与高亮协议
- 不提供每种连接模式各自独立的亮度配置
- 不支持亮度为 0% 的完全熄屏模式

## 现状问题

当前固件在启动时固定调用 `setBrightness8(64)`，网页端没有亮度控制能力。用户无法按环境亮度即时调整屏幕显示，也无法让设备记住自己的亮度偏好。现有高亮同步已经支持「即时推送」交互，但亮度没有对应的控制接口。

## 方案

### 1. 网页交互

- 亮度控件放在设置弹窗内，位于「背景色（透明区域）」下方、「操作」区域上方
- 使用单个滑杆输入，范围为 `10% - 100%`
- 滑杆右侧显示当前亮度百分比，例如 `62%`
- 当页面没有当前目标设备时，滑杆禁用，并展示提示文案
- 用户拖动滑杆时立即同步；为避免在连续拖动时发送过密，前端对发送动作增加短时间节流

### 2. 前端状态模型

- `window.appState` 增加以下状态：
  - `brightnessPercent`：当前 UI 展示的亮度百分比
  - `deviceBrightnessLoaded`：是否已经从设备读取过一次真实亮度
  - `brightnessSyncTimer`：滑杆拖动时的短时节流计时器
  - `isSyncingBrightness`：防止亮度请求并发重入
- 页面初始化时先从 `localStorage` 读取上次保存的亮度百分比，保证 UI 有初始值
- 一旦蓝牙连接成功、打开设置弹窗，或切换到已记住的 WiFi 设备时，主动向设备读取当前亮度，再回填滑杆，避免本地缓存与设备真实值不一致

### 3. 前端发送策略

- 新增 `sendBrightnessToESP32()`，优先级与高亮同步一致：
  - 已连接 BLE：直接通过 Web Bluetooth 发送
  - 未连接 BLE 且当前模式为 WiFi：走服务端 `/api/wifi/brightness`
  - 无目标设备：不发送，仅保留 UI 值
- 拖动滑杆时只同步亮度，不触发图像重发
- 手动点击「发送到 ESP32」时，不额外重复发送亮度；亮度独立维护，图像与亮度解耦

### 4. BLE 协议设计

- 在现有 BLE 命令特征上新增两条命令：
  - `0x09`：设置亮度，格式为 `[0x09][brightness]`
  - `0x0A`：读取亮度，格式为 `[0x0A]`
- 设备返回亮度通知，格式为 `[0x26][brightness]`
- `brightness` 使用 1 字节原始亮度值，范围固定为 `26 - 255`
- 网页端显示百分比，设备端使用原始值；二者之间由前端做双向换算

### 5. WiFi 备选接口设计

- 固件 HTTP 服务新增亮度接口：
  - `POST /brightness`：请求体为 1 字节亮度值，设备立即应用并持久化，响应 `{"success":true,"brightness":<value>}`
  - `GET /brightness`：返回 `{"success":true,"brightness":<value>}`
- 服务端新增转发接口：
  - `POST /api/wifi/brightness`
  - `GET /api/wifi/brightness?device_uuid=...`
- WiFi 路径的设备定位仍然只依赖已记住的 `targetDeviceUuid`

### 6. 固件设计

- 在 `firmware/src/main.cpp` 增加统一亮度管理入口：
  - `loadBrightness()`
  - `applyBrightness(uint8_t value, bool persist)`
  - `getBrightness()`
- 亮度持久化使用 Arduino `Preferences`
- NVS 统一使用命名空间 `beadcraft`、键名 `brightness`
- 启动流程：
  1. 初始化 `Preferences`
  2. 从 NVS 读取亮度
  3. 若无保存值，则使用默认亮度
  4. 调用 `dma_display->setBrightness8(...)`
- 收到新的亮度命令后：
  1. 将亮度值钳制到 `26 - 255`
  2. 立即调用 `setBrightness8(...)`
  3. 在 `persist = true` 时写入 NVS
  4. 返回当前亮度通知或 HTTP 响应

### 7. 数值映射

- 网页显示范围：`10% - 100%`
- 固件原始范围：`26 - 255`
- 前端换算规则：
  - 百分比转设备值：线性映射到 `26 - 255`
  - 设备值转百分比：反向线性映射到 `10% - 100%`
- 网页端必须保证：
  - 不向设备发送小于 `26` 的亮度值
  - 不显示低于 `10%` 的滑杆值

### 网页端

  - 新增亮度滑杆、数值展示与提示容器
- `static/app.js`
  - 新增亮度状态
  - 新增 BLE / WiFi 亮度查询与同步方法
  - 在设置弹窗显示、BLE 连接成功、WiFi 目标可用时刷新设备亮度
- `static/i18n.js`
  - 增加亮度相关中英文文案

### 服务端

- `main.py`
  - 新增 WiFi 亮度查询与设置转发接口
  - 统一错误提示与设备未注册处理

### 固件端

- `firmware/src/main.cpp`
  - 新增亮度加载、应用与持久化逻辑
- `firmware/lib/ble-receiver/BLEImageReceiver.h`
  - 新增亮度命令解析与亮度通知
  - 新增 WiFi `/brightness` 处理
- `main.cpp` 作为亮度状态的唯一所有者，通过构造参数向 `BLEImageReceiver` 注入 `setBrightness` 与 `getBrightness` 回调，避免在多个接收器中各自维护亮度状态

## 数据流

### BLE 亮度同步

1. 用户打开设置弹窗
2. 页面检测到已有 BLE 连接，发送 `0x0A` 读取亮度
3. 固件返回 `[0x26][brightness]`
4. 页面回填滑杆与百分比
5. 用户拖动滑杆
6. 页面节流后发送 `[0x09][brightness]`
7. 固件立即应用并写入 NVS
8. 固件回发当前亮度通知
9. 页面以通知值作为最终状态

### WiFi 亮度同步

1. 页面已记住目标设备序列号
2. 用户切换到 WiFi 模式并打开设置
3. 页面请求 `/api/wifi/brightness`
4. 服务端转发到设备 `GET /brightness`
5. 设备返回当前亮度
6. 用户拖动滑杆
7. 页面节流后请求 `/api/wifi/brightness`
8. 服务端转发到设备 `POST /brightness`
9. 设备应用亮度并写入 NVS

## 错误处理

- 无目标设备时，滑杆保持禁用，不发送请求
- BLE 已选择但未真正连接时，不主动弹出设备选择器；保持与当前自动发送策略一致
- BLE 亮度设置失败时，只提示 toast，不影响当前页面图像与高亮状态
- WiFi 设备未注册或离线时，沿用现有 WiFi 错误提示
- 固件收到非法亮度值时，钳制到合法区间，不返回异常值
- 如果亮度读取失败，网页保留本地缓存值并提示读取失败

## 默认值

- 固件默认亮度沿用当前近似体验，设置为 `64`
- 网页默认显示百分比按 `64` 对应的百分比换算
- 当设备首次写入亮度后，以 NVS 中的值为准

## 验证策略

### 网页端

- 打开设置弹窗时能看到亮度滑杆与当前百分比
- 已连接 BLE 时，拖动滑杆会即时发送亮度同步
- 切换到 WiFi 模式且已有记住设备时，亮度也可设置
- 没有目标设备时，滑杆禁用
- 重新打开页面后，滑杆能先显示缓存值；连接设备后会被真实值覆盖

### 固件端

- 固件首次启动时使用默认亮度
- 亮度变更后无需重启即可生效
- 设备重启后保持上次亮度
- BLE 与 WiFi 两条链路的亮度行为一致
- 亮度值始终落在 `26 - 255`

## 风险

- BLE 命令通知与现有 ACK / WiFi 扫描通知共用特征，新增亮度通知时需要避免与现有包类型冲突
- 滑杆拖动过于频繁时，若节流窗口过小，可能造成 BLE 写入拥塞
- WiFi 接口新增 `GET /brightness` 和 `POST /brightness` 后，需要确保不会阻塞现有 `/image` 与 `/highlight` 处理
- 固件亮度持久化若每次拖动都立即写 NVS，理论上会增加写入次数；需要通过节流将写入频率控制在可接受范围
