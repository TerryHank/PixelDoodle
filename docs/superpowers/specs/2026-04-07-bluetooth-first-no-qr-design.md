# 蓝牙优先无扫码改造设计

**日期：** 2026-04-07

## 目标

将网页端的主连接路径改为蓝牙设备列表直连，移除扫码与串口相关入口；保留 WiFi 作为备选发送路径，但只对已通过蓝牙记住序列号的同一台设备生效。同步调整固件待机页与高亮显示逻辑。

## 范围

本次改造包含以下内容：

- 网页首页移除扫码入口，主入口直接改为上传文件
- 网页设置面板移除串口模式，只保留蓝牙与 WiFi
- 网页移除二维码扫描相关入口、弹窗与引导文案
- 蓝牙连接改为直接从设备列表选择 `BeadCraft-*` 设备
- WiFi 模式只使用已记住的设备序列号，不再依赖二维码锁定设备
- 色板高亮同步恢复为黑色按黑色发出
- 固件待机页不再显示二维码，改为在屏幕中间显示序列号
- 固件高亮模式下，所有命中的颜色统一以蓝灯显示，不保留原色

## 非目标

本次不做以下内容：

- 不重做现有页面视觉风格
- 不重构后端 API 结构
- 不移除后端现有串口与扫码相关接口
- 不改变整图发送时的颜色数据

## 现状问题

当前网页默认流程依赖二维码锁定设备，首页上传区、顶部工具栏、设置弹窗和 WiFi 备选链路都与扫码流程耦合。串口模式仍在前端保留，但新的使用口径已经转为蓝牙优先。固件待机页仍显示二维码，高亮模式命中后显示原像素色，无法满足统一蓝灯高亮的目标。

## 方案

### 1. 网页连接模型

- 默认连接模式保留为蓝牙
- 蓝牙模式下，用户通过现有蓝牙设备列表与连接按钮直接选择 `BeadCraft-*` 设备
- 蓝牙连接成功后，从设备名中解析 12 位序列号并记入 `targetDeviceUuid`
- WiFi 模式不再要求扫码锁定设备，只读取已经记住的 `targetDeviceUuid`
- 如果没有已记住的序列号，WiFi 模式只展示提示，不允许发送

### 2. 网页首页与设置面板

- 顶部工具栏移除扫码按钮
- 首页上传区保持原有组件风格，但文案与点击行为改为直接触发文件选择
- 设置面板移除串口模式按钮与串口配置区
- WiFi 面板保留为备选模式，展示当前已记住设备
- 若需要从蓝牙侧为设备配置 WiFi，继续沿用已存在的 BLE WiFi 能力，但入口不再通过扫码弹窗触发

### 3. 色板高亮同步

- 前端删除“黑色高亮改发白色”的特殊映射
- 色板高亮同步发送时，所有颜色按自身 RGB 值发送
- 高亮协议格式保持不变，仍使用现有 `0x04 + count + RGB565[]`

### 4. 固件待机页

- 删除二维码绘制流程与二维码相关文案
- 保留序列号生成逻辑
- 待机页在 64x64 面板中心显示序列号
- 继续通过串口日志输出设备序列号，便于调试

### 5. 固件高亮显示

- 串口接收库与 BLE 接收库都保留“按 RGB565 值命中高亮颜色”的匹配逻辑
- 一旦命中高亮颜色，不再显示原像素色，统一显示固定蓝色 `RGB565 0x001F`
- 未命中的像素保持背景黑色

## 组件影响

### 网页端

- `templates/index.html`
  - 删除扫码按钮、扫码弹窗、串口设置区
  - 调整上传区默认文案与行为
- `static/app.js`
  - 移除扫码依赖的初始化与 UI 流转
  - 只保留蓝牙 / WiFi 两种前端连接模式
  - 蓝牙连接成功后记住设备序列号
  - WiFi 使用记住的设备序列号
  - 恢复高亮发送颜色原样传递
- `static/i18n.js`
  - 移除或替换扫码、串口相关文案
- `static/style.css`
  - 删除扫码弹窗与扫码引导样式
  - 保持上传区和设置区的原有视觉基调

### 固件端

- `firmware/src/main.cpp`
  - 待机页入口改为显示序列号
  - 去除配对 URL 与二维码待机调用
- `firmware/lib/beadcraft-receiver/BeadCraftReceiver.h`
  - 删除二维码绘制相关逻辑
  - 增加居中显示序列号的方法
  - 串口高亮模式命中后统一显示蓝色
- `firmware/lib/ble-receiver/BLEImageReceiver.h`
  - BLE 高亮模式命中后统一显示蓝色

## 数据流

### 蓝牙主路径

1. 用户打开设置面板
2. 选择蓝牙模式
3. 点击连接按钮，从系统蓝牙选择器中选择 `BeadCraft-*` 设备
4. 页面解析设备名中的序列号并写入本地状态
5. 生成图案后通过 Web Bluetooth 直接发送图像或高亮命令

### WiFi 备选路径

1. 用户曾通过蓝牙成功连接设备，页面已记住序列号
2. 用户切到 WiFi 模式
3. 页面读取已记住的序列号并作为唯一目标设备
4. 生成图案后通过服务端 WiFi 中继发送到该设备

## 错误处理

- 蓝牙未连接时，上传与生成功能不阻塞，但发送时明确提示需要先连接设备
- WiFi 模式下若没有记住的设备序列号，页面提示“请先通过蓝牙连接设备”
- WiFi 发送失败时，保留当前已有错误提示
- 高亮命令发送失败时继续只报 toast，不中断页面编辑状态

## 验证策略

### 网页端

- 打开首页时，不再出现扫码入口或扫码弹窗入口
- 上传区点击直接触发文件选择
- 设置面板只显示蓝牙与 WiFi 两种模式
- 蓝牙连接后能记住设备序列号
- WiFi 模式只对已记住设备可用
- 选中黑色时，高亮同步发送仍然是黑色值

### 固件端

- 固件启动后，屏幕中间显示序列号，不再显示二维码
- 发送高亮命令后，命中颜色全部显示为蓝色
- 串口与 BLE 两种高亮路径行为一致

## 风险

- `static/app.js` 现有扫码逻辑有重复定义，删改时需要避免留下悬挂引用
- WiFi 备选链路原先与二维码锁定设备耦合，改造时需要确保“记住的序列号”成为唯一数据源
- 固件待机页移除二维码后，若文字布局计算不当，可能导致序列号不居中或超界
